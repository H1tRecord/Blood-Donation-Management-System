================================================================================
CARDINALITY LEGEND
================================================================================

1:1  = One-to-One Relationship
      One entity in table A relates to exactly one entity in table B

1:N  = One-to-Many Relationship
      One entity in table A can relate to many entities in table B

N:M  = Many-to-Many Relationship
      Many entities in table A can relate to many entities in table B

KEY SYMBOLS:
PK   = Primary Key (uniquely identifies each record)
FK   = Foreign Key (references primary key in another table)

================================================================================
KEY RELATIONSHIPS SUMMARY
================================================================================

1. USER (1) ──has──> (1) DONOR/STAFF/ADMIN (Inheritance Pattern)
   - One user account for authentication
   - User role determines access level

2. DONOR (1) ──donates──> (N) DONATION_RECORD
   - One donor can have many donation records
   - Tracks donation history

3. DONATION_RECORD (1) ──includes──> (1) HEALTH_METRICS
   - Health metrics captured during each donation

4. DONOR (1) ──has──> (N) HEALTH_METRICS
   - Multiple health records per donor over time

5. DONOR (1) ──books──> (N) APPOINTMENT
   - One donor can have multiple appointments

6. APPOINTMENT (N) ──at──> (1) DONATION_CENTER
   - Multiple appointments at one center

7. DONATION_CENTER (1) ──manages──> (N) BLOOD_INVENTORY
   - Each center tracks inventory for each blood type

8. APPOINTMENT (1) ──logged_by──> (N) APPOINTMENT_STATUS_LOG
   - Status changes tracked for audit trail

9. STAFF (1) ──processes──> (N) DONATION_RECORD
   - Staff member processes donations

10. STAFF (1) ──records──> (N) HEALTH_METRICS
    - Staff member records health metrics

================================================================================
ENTITY CARDINALITY NOTATION
================================================================================

1:1  = One to One
1:N  = One to Many
N:M  = Many to Many

PK   = Primary Key
FK   = Foreign Key

================================================================================
DATA INTEGRITY CONSTRAINTS
================================================================================

1. DONATION_RECORD:
   - nextEligibleDate = lastDonationDate + 56 days (for whole blood)
   - Status transition: Pending → Confirmed → Checked-In → Completed/Cancelled

2. BLOOD_INVENTORY:
   - Status alerts when unitCount falls below thresholds
   - Automatic status updates: GREEN (20+), YELLOW (10-19), RED (<10)

3. APPOINTMENT:
   - Cannot book appointment if donor is not eligible
   - Max 1 appointment per time slot per center (enforced at application level)
   - Cannot double-book same donor

4. HEALTH_METRICS:
   - Hemoglobin: Valid range 7-20 g/dL
   - Blood Pressure: Format validation SYS/DIA (e.g., 120/80)

5. USER/SESSION:
   - Automatic session expiry after 30 minutes of inactivity
   - Only one active session per user (optional feature)
