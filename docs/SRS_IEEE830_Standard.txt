1. Introduction
1.1  Purpose
This Software Requirements Specification (SRS) document provides a comprehensive technical
description of the Blood Donation Management System (BDMS). Its primary purpose is to
establish a clear agreement between project stakeholders, medical staff, and developers by
defining the functional and non-functional requirements of the application. This system serves
as a central coordination hub for managing blood donation appointments, monitoring inventory
levels in real-time, and identifying eligible donors to maintain a stable blood supply. Now
integrated with Google Firebase for secure data persistence.

1.2  Document Conventions
This document adheres to the IEEE 830-1998 standard for software requirements. To ensure
technical precision and clarity for developers, testers, and stakeholders, the following structural
and typographical conventions are employed:

1.2.1 Terminology and Obligation
The following keywords are used to denote the priority and mandatory nature of specific
requirements:
●  "Shall": Represents a mandatory, high-priority requirement essential for the system's core
   functionality.
●  "Should": Represents a recommended or desirable requirement that adds significant value
   but is not critical for initial deployment.
●  "May": Represents an optional or low-priority requirement for future consideration.

1.2.2 Requirement Identifiers
For traceability and cross-referencing throughout the development lifecycle, each requirement is
assigned a unique identifier:
●  [FR-00]: Functional Requirements (e.g., [FR-01], [FR-02]).
●  [NFR-00]: Non-Functional Requirements (e.g., [NFR-01], [NFR-02]).

1.2.3 Structural Conventions
The document utilizes specific listing formats to organize information based on logical
relationships:
●  Numbered Lists: Used when the sequence or order of items is critical, such as in
   step-by-step procedures, chronological events, or prioritized rankings.
●  Bullet Points (Primary): Used for grouping related items of equal importance where the
   specific order does not change the meaning or outcome (unordered lists).
○  Secondary Bullet Points: Used for hierarchical decomposition providing sub-details or
   secondary constraints belonging to a primary bullet point.

1.3  Intended Audience and Reading Suggestions
This document is designed for the following stakeholders:
●  Project Managers: To verify that the proposed solution meets business objectives. They
   should focus on Section 1.4 (Scope) and Section 5 (Nonfunctional Requirements).
●  Software Developers: To understand the technical specifications for implementing the
   React-based frontend and associated Firebase data layers. They should prioritize Section 3
   (External Interfaces) and Section 4 (System Features).
●  Quality Assurance (QA) Testers: To derive test cases and validate that the system
   performs according to the specified accuracy and performance metrics. They should use
   Section 5 and Section 4 to derive test cases.
●  Medical Staff: To understand the operational workflows for appointment and inventory
   management. They should focus on Section 2.2 and Section 4.

1.4  Product Scope
The Blood Donation Management System is a web-based application designed to streamline the
entire lifecycle of blood donation, from donor registration to inventory tracking.

1.4.1 Included Capabilities (In-Scope):
●  User Management: Registration, authentication, and profile management for Donors, Staff,
   and Administrators via Firebase Auth.
●  Appointment Lifecycle: A guided two-step booking process for donors and a calendar-based
   management interface for staff.
●  Inventory Monitoring: Real-time tracking of blood units by type, including automated
   expiry alerts and manual batch adjustments.
●  Eligibility Logic: Automated calculation of the 56-day deferral period based on donation
   history stored in Firebase RTDB.

1.4.2 Exclusions (Out-of-Scope):
●  Laboratory Equipment: The system does not interface with physical blood analysis or
   collection hardware.
●  External Integration: Direct integration with external hospital EHR systems is excluded
   from the current version.
●  Notifications: SMS notifications are not supported; communication is limited to in-app
   alerts and manual contact facilitation.

2. Overall Description
2.1  Product Perspective
The Blood Donation Management System (BDMS) is an independent, web-based platform designed
to centralize and digitize blood bank operations. It functions as a single-page application (SPA)
integrated with Firebase as a backend-as-a-service (BaaS) for authentication and data persistence.

2.1.1 System Architecture
The application follows a modern serverless architecture:
●  Frontend Layer: Developed with React 19.x, providing a reactive and component-based
   user interface.
●  Backend Layer: Firebase Authentication manages user security and sessions.
●  Data Layer: Firebase Realtime Database (RTDB) provides consistent, real-time data
   synchronization for all users.
●  Routing Layer: Implements React Router v6 for client-side navigation and role-based
   access protection.

2.2  Product Functions
The BDMS core functions are categorized into automated tracking and management tools:

2.2.1 Core Tracking Functions:
1.  Dynamic Eligibility Calculation: Automated monitoring of donor eligibility status using
    a 56-day deferral algorithm applied to live historical data.
2.  Stock Health Visualizer: Real-time inventory monitoring with color-coded alerts for
    Critical (<10 units), Low (10-19 units), and Adequate (20+ units) supply levels.
3.  Expiry Management: Automated identification of blood units expiring within a 7-day
    window to prevent medical waste.

2.2.2 Operational Management:
1.  Appointment Scheduling: Multi-step chronological booking process for donors and
    calendar-based timeline for staff clinical planning.
2.  User Access Control: Centralized administration for role-based account management
    (activation/deactivation) and record search.

2.3  User Classes and Characteristics
The system supports three distinct user classes with specialized permissions:
●  Registered Donors: End-users who utilize the system for registration, eligibility tracking,
   and appointment booking. They require a high-usability, low-friction interface.
●  Medical Staff: Clinical professionals responsible for processing donations, verifying blood
   types, and managing daily inventory levels. They require high efficiency and data accuracy.
●  Administrators: Technical staff responsible for overseeing all system users and managing
   account security states.

2.4  Operating Environment
2.4.1 Client Browser Environment
The BDMS executes within standard modern web browsers on mobile or desktop devices:
●  Desktop: Chrome 90+, Firefox 88+, Safari 14+, or Edge 90+.
●  Mobile: iOS Safari 14+ or Android Chrome 90+.
●  Required Client Features: JavaScript enabled; HTTPS support for Firebase SDK.

2.4.2 Build and Development Environment
●  Node.js: Version 18.x or higher for local development.
●  Bundler: Vite 7.x for optimized resource packaging and HMR (Hot Module Replacement).
●  Backend: Firebase Project (Admin Console) for environment configuration.

2.5  Design and Implementation Constraints
1.  Persistent Cloud Storage: The application shall maintain all session and operational data
    in Firebase, ensuring data sharing and consistency across different devices.
2.  Standard Deferral: The system shall strictly enforce a 56-day whole blood deferral period in
    accordance with AABB standards.
3.  Asynchronous Operations: UI components shall handle data fetching asynchronously to
    provide a smooth user experience despite network latency.

2.6  User Documentation
Users shall be supported by the following documentation:
●  BDMS User Guide: Digital documentation detailing the appointment booking lifecycle for
   donors.
●  Clinical Staff Workbook: Detailed procedures for inventory adjustment, donor search,
   and donation processing.
●  On-Screen Guidance: Contextual tooltips and data validation labels within the UI.

2.7  Assumptions and Dependencies
Project Assumptions
●  Accurate History: It is assumed that donors provide truthful information regarding their
   previous donation dates.
●  Standard Deferral: It is assumed that 56 days is the correct and only deferral period
   required for the localized center operations.
●  Cloud Connectivity: It is assumed that users have stable internet access to communicate
   with Firebase services.

Technical Dependencies
●  React Library: Dependency on the stable APIs of React 19 and React Router v6.
●  Firebase SDK: Dependency on Firebase Authentication and Realtime Database modules.
●  Vite Runtime: Dependency on the Vite development server for local execution and bundling.

3. External Interface Requirements
3.1  User Interfaces
The BDMS user interfaces are developed as high-fidelity React components with a focus on
operational efficiency and accessibility.

3.1.1 General Design Principles
●  Consistency: All portals (Donor, Staff, Admin) shall share a unified color palette and
   component library.
●  Responsiveness: The UI shall utilize liquid layouts (CSS Grid/Flexbox) to support devices
   ranging from 320px (mobile) to 4K (desktop).
●  State Feedback: All destructive actions (e.g., deleting a unit) shall require user
   confirmation via modal dialogs.

3.1.2 Specific Interface Requirements
1.  Donor Dashboard: Shall provide a clear "Eligibility Countdown" if the donor is within the
    56-day deferral window.
2.  Inventory Table: Shall use color-coded rows (Red-Critical, Yellow-Low, White-Adequate)
    to signify stock health.
3.  Admin Panel: Shall allow administrators to filter the user list by Name, Email, or Role.

3.2  Hardware Interfaces
The BDMS is a software-only product that requires standard human-interface devices:
●  Pointing Device: Mouse or touch-screen for navigation and selection.
●  Keyboard: QWERTY or virtual keyboard for data entry.
●  Display: Standard RGB monitor or mobile display with minimum 320px width.

3.3  Software Interfaces
3.3.1 Browser Runtime
The system interface is dependent on the following modern web technologies:
●  Firebase SDK API: For real-time database and auth operations.
●  DOM Level 3: For reactive UI updates via React's virtual DOM.
●  ECMAScript 2022 (ES13): For modern JavaScript functionality (Optional Chaining, etc.).

3.4  Communications Interfaces
3.4.1 HTTPS Protocol
All communication between the client browser and Firebase servers shall
be encrypted using Transport Layer Security (TLS) 1.2 or higher.

3.4.2 API Standards
Data exchange between the client and Firebase shall follow the Firebase Realtime
Database wire protocol utilizing JSON (JavaScript Object Notation).

B) INTERNAL DEPENDENCIES
   - Authentication system must be operational before any other feature
   - Donation history data must be accurate for eligibility calculation
   - Appointment availability depends on donation center configuration
   - Inventory accuracy depends on complete donation processing

C) SEQUENTIAL DEPENDENCIES
   - User registration must precede appointment booking
   - Blood type must be recorded before donation completion
   - Donation must be processed before inventory adjustment
   - Appointment must be created before status tracking

4. System Features
================================================================================

4.1  User Management and Authentication
This feature set manages the lifecycle of user identities and their access to the BDMS portals.

4.1.1 Requirement: User Registration
[FR-01] The system shall provide a registration interface for new donors and staff members.
[FR-02] The registration form shall mandate the following fields: Full Name, Email, Phone,
      and Password.
[FR-03] The system shall enforce unique email constraints to prevent duplicate account
      creation.
[FR-04] The system shall explicitly NOT request blood type during registration, displaying
      a mandatory notice that "Blood type is determined by medical staff during the first
      donation."
[FR-05] Upon successful registration, the system shall generate a unique persistent ID and
      persist the record to the Firebase Realtime Database.

4.1.2 Requirement: Secure Authentication
[FR-06] The system shall provide a centralized login interface requiring Email and Password
      credentials.
[FR-07] The system shall validate credentials against Firebase Authentication and
      redirect users to their role-specific dashboard (Donor, Staff, or Admin) upon
      success.
[FR-08] The system shall maintain session state using context providers, clearing all
      session data upon a manual "Logout" action.
[FR-09] The system shall display clear error notifications for "Invalid Email/Password" or
      "Incomplete Fields."

4.1.3 Requirement: Role-Based Access Control (RBAC)
[FR-10] The system shall restrict access to the /admin/* routes to users with the 'admin'
      role.
[FR-11] The system shall restrict access to the /staff/* routes to users with the 'staff'
      role.
[FR-12] Unauthorized access attempts to protected routes shall automatically redirect the
      user to the Login page.

4.2  Donor Dashboard and Clinical Processing
This feature set provides donors with their clinical status and history.

4.2.1 Requirement: Dynamic Eligibility Tracking
[FR-13] The system shall calculate a donor's eligibility status based on a strict 56-day
      deferral period from their most recent 'Completed' donation.
[FR-14] Eligible Donors: The system shall display a green "Eligible" status and enable the
      "Book Appointment" functionality.
[FR-15] Ineligible Donors: The system shall display a red "Ineligible" status and a
      countdown timer indicating the exact days remaining until the next eligible date.
[FR-16] The system shall automatically block appointment booking attempts for donors
      flagged as "Ineligible."

4.2.2 Requirement: Donation History Timeline
[FR-17] The system shall display a chronological list of all past donation events for the
      authenticated donor.
[FR-18] Each history entry shall include the Date, Center Location, and Status (Completed/
      Cancelled) of the donation.

4.3  Appointment Management Workflow
This feature set coordinates the scheduling of donation slots.

4.3.1 Requirement: Guided Appointment Booking
[FR-19] The system shall provide a multi-step booking wizard for eligible donors.
[FR-20] Step 1 (Selection): The user shall select a preferred Date and available Time Slot
      from the clinical schedule.
[FR-21] Step 2 (Confirmation): The system shall present a summary of the selection for
      final user approval.
[FR-22] The system shall update the global appointment collection in Firebase RTDB
      immediately upon successful booking.

4.3.2 Requirement: Staff Clinical Calendar
[FR-23] The system shall provide a calendar-based interface for medical staff to view
      daily appointment density.
[FR-24] Staff shall be able to update appointment statuses in real-time (e.g., marking
      a donor as "Checked-In" or "Donation Completed").
[FR-25] Marking an appointment as "Completed" shall automatically trigger an update to
      the donor's medical history and the inventory stock levels.

4.4  Inventory and Stock Health
This feature set manages the physical blood supplies.

4.4.1 Requirement: Real-Time Stock Visualization
[FR-26] The system shall maintain a categorized inventory count for all eight primary
      blood types (A+, A-, B+, B-, AB+, AB-, O+, O-).
[FR-27] The system shall apply visual health indicators based on the following logic:
      - Critical: <10 units (Red)
      - Low: 10-19 units (Yellow)
      - Adequate: 20+ units (Green/White)

4.4.2 Requirement: Batch Management and Expiry
[FR-28] Staff shall have the capability to manually adjust stock levels through "Update
      Inventory" modals.
[FR-29] The system shall track expiry dates for individual batches, flagging any units
      expiring within the next 7 calendar days.

4.5  Administrative Oversight
This feature set provides global system control.

4.5.1 Requirement: User Account Administration
[FR-30] Administrators shall have access to a master list of all system users.
[FR-31] The system shall provide functionality to "Activate" or "Deactivate" any user
      account, effectively blocking or granting system access.
[FR-32] Administrators shall be able to edit core user profile details, including Role
      and Contact Information.

5. Nonfunctional Requirements
5.1  Performance Requirements
[NFR-01] Response Time: The system shall render UI transitions (routing) in less than
       300ms under standard network conditions.
[NFR-02] Data Synchronization: Changes to Firebase RTDB shall reflect on all connected
       clients within 2 seconds.
[NFR-03] Concurrency: The application shall support multiple simultaneous users tracking
       live inventory and appointment updates.

5.2  Safety and Security Requirements
[NFR-04] Access Control: The system shall enforce role-based redirection, ensuring that
       unauthenticated users cannot view sensitive clinical or administrative data.

5.5 BUSINESS RULES

BR1: DONATION ELIGIBILITY
     Blood donors must wait 56 days between whole blood donations. No donor can
     schedule or complete a donation if less than 56 days have passed since their
     last whole blood donation.

BR2: BLOOD TYPE DETERMINATION
     Blood type is not collected during donor registration. Blood type must be
     determined by qualified medical staff during the first blood donation
     appointment based on laboratory testing.

BR3: APPOINTMENT AVAILABILITY
     Appointment slots can only be booked during donation center operating hours.
     Each time slot is limited to one donor appointment maximum unless the staff
     specifically increases capacity.

BR4: INVENTORY MANAGEMENT
     Blood inventory levels must be monitored continuously. When inventory for
     any blood type drops below 10 units (critical level), the system shall
     alert staff that new donations or supplies are urgently needed.

BR5: DONATION CENTER OPERATIONS
     The donation center operates during scheduled hours and accepts appointment
     bookings during those hours. Staff must update center operating hours if
     the center will be temporarily closed.

BR6: SESSION SECURITY
     User sessions automatically expire after 30 minutes of inactivity for
     security. Users must re-authenticate to continue using the system.

BR7: DATA RETENTION
     Donor records and donation history are retained for a minimum of 5 years
     for medical and legal compliance. Records must be securely archived after
     this period.

BR8: APPOINTMENT STATUS WORKFLOW
     Appointments follow a strict status workflow: Pending → Confirmed → 
     Checked-In → Completed, or may be Cancelled at any point. Appointments
     cannot skip steps in this workflow.

BR9: PRIVACY AND CONFIDENTIALITY
      Donor personal information and donation records are private and visible
      only to the donor themselves and authorized medical staff. No other
      users or systems can access this information without explicit consent.

1. Donor enters registration form
2. System validates input

6. System redirects to login

1. Donor verifies eligibility
2. Donor selects donation center
3. Donor selects date and time
4. System checks availability
5. Donor confirms booking
6. System creates appointment record
7. System sends confirmation email (Phase 2)
8. System redirects to dashboard

1. Staff views appointment calendar
2. Staff marks donor as checked-in
3. Staff verifies donor blood type information
4. Staff marks donation as completed
5. System creates donation record
6. System calculates next eligible date
7. System decrements inventory
8. System sends completion notification (Phase 2)
