

1. Introduction

1.1 Purpose
This Software Requirements Specification (SRS) document provides a comprehensive technical description of the Blood Donation Management System (BDMS). Its primary purpose is to establish a clear agreement between project stakeholders, medical staff, and developers by defining the functional and non-functional requirements of the application. This system serves as a central coordination hub for managing blood donation appointments, monitoring inventory levels in real-time, and identifying eligible donors to maintain a stable blood supply.

1.2 Document Conventions
This document adheres to the IEEE 830-1998 standard for software requirements. To ensure technical precision and clarity for developers, testers, and stakeholders, the following structural and typographical conventions are employed:

1.2.1 Terminology and Obligation
The following keywords are used to denote the priority and mandatory nature of specific
requirements:
"Shall": Represents a mandatory, high-priority requirement essential for the system's core functionality.
"Should": Represents a recommended or desirable requirement that adds significant value but is not critical for initial deployment.

1.4 Product Scope
The Blood Donation Management System is a web-based application designed to streamline the entire lifecycle of blood donation, from donor registration to inventory level tracking.

1.4.1 Included Capabilities (In-Scope):
User Management: Registration, authentication, and profile management for Donors, Staff, and Administrators.
Appointment Lifecycle: A guided two-step booking process for donors and a calendar-based management interface for staff.
Inventory Monitoring: Real-time tracking of blood units by type, including automated expiry alerts and manual batch adjustments.
Eligibility Logic: Automated calculation of the 56-day deferral period based on donation history.

1.4.2 Exclusions (Out-of-Scope):
Advanced Blood Inventory Logic: The system will not implement a full blood inventory management system. It will only provide basic monitoring of blood supply levels rather than comprehensive inventory tracking and management..
Donor Screening and Medical Processing: The system does not handle medical screening or testing, as these procedures remain under the responsibility of medical professionals and existing clinical workflows.
Notifications: SMS notifications are not supported; communication is limited to in-app alerts and manual contact facilitation.

2. Overall Description

2.1 Product Perspective
The Blood Donation Management System (BDMS) is an independent, web-based platform designed to centralize and digitize blood bank operations. It functions as a standalone single-page application (SPA) that coordinates the relationship between donors, medical staff, and stock level management.

2.2 Product Functions
The BDMS core functions are categorized into automated tracking and management tools:

Core Tracking Functions:
Dynamic Eligibility Calculation: Automated monitoring of donor eligibility status using a 56-day deferral algorithm applied to historical data.
Stock Health Visualizer: Real-time inventory monitoring with color-coded alerts for Critical (<10 units), Low (10-19 units), and Adequate (20+ units) supply levels.
Expiry Management: Automated identification of blood units expiring within a 7-day window to prevent medical waste.

Operational Management:
Appointment Scheduling: Multi-step chronological booking process for donors and calendar-based timeline for staff clinical planning.
User Access Control: Centralized administration for role-based account management (activation/deactivation) and record search.

2.3 User Classes and Characteristics
The system supports three distinct user classes with specialized permissions:
Registered Donors: End-users who utilize the system for registration, eligibility tracking, and appointment booking. They require a high-usability, low-friction interface.
Medical Staff: Clinical professionals responsible for processing donations, verifying blood types, and managing daily inventory levels. They require high efficiency and data accuracy.
Administrators: Technical staff responsible for overseeing all system users and managing account security states.

2.4 Operating Environment
The BDMS executes within standard modern web browsers:

Desktop: Chrome 90+, Firefox 88+, Safari 14+, or Edge 90+.
Required Client Features: JavaScript enabled; HTTPS support for Firebase SDK.

2.5 Design and Implementation Constraints
Stateless Persistence: The application shall maintain all session and operational data locally within the browser's localStorage, limiting data sharing across different devices.
Standard Deferral: The system shall strictly enforce a 56-day whole blood deferral period in accordance with AABB standards.
Modular Components: Code shall follow a modular React architecture to ensure maintainability for future backend integration.

2.6  User Documentation
Users shall be supported by the following documentation:
BDMS User Guide: Digital documentation detailing the appointment booking lifecycle for donors.
Clinical Staff Workbook: Detailed procedures for inventory adjustment, donor search, and donation processing.
On-Screen Guidance: Contextual tooltips and data validation labels within the UI.

2.7 Assumptions and Dependencies
Project Assumptions
Accurate History: It is assumed that donors provide truthful information regarding their previous donation dates.
Standard Deferral: It is assumed that 56 days is the correct and only deferral period required for the localized center operations.

Technical Dependencies
React Library: Dependency on the stable APIs of React 19 and React Router v6.
Browser LocalStorage: Dependency on persistent local storage not being cleared by browser privacy settings between sessions.

3. External Interface Requirements

3.1 User Interfaces
The Blood Donation Management System user interface shall be designed for high responsiveness and accessibility, following modern web design principles and accessibility standards. The system shall maintain consistent design language across all interfaces while optimizing for different user needs and device types.

3.1.1 Donor User Interface
The donor interface focuses on ease of use for account management and appointment scheduling.

A) AUTHENTICATION INTERFACE
Login Screen:
Email address input field with validation
Password input field with character masking
"Login" button
"Create New Account" link to registration

Registration Screen:
Full Name input (Required, max 100 characters).
Email Address input field with format validation
Password input field (required, minimum 6 characters)
"Create Account" button

B) DONOR DASHBOARD
Navigation: Global header containing donor name and logout functionality.
Profile Card: Summary of blood type, email, and total lifetime donations.
Eligibility Tracker: Dynamic status message or countdown timer indicating when the donor can next donate.
Donation History: A chronological timeline of all past successful donations.

C) APPOINTMENT BOOKING INTERFACE
Scheduler: Interactive 7-day calendar with real-time slot availability.
Confirmation View: Summary displaying donor name, blood type, and booking terms.
Controls: "Confirm Booking" to finalize, "Go Back" to modify, and "Cancel" to abandon the flow.

3.1.2 Staff User Interface
The staff interface is optimized for data density and quick operational actions.

A) Staff Dashboard
Overview Statistics: High-level metric cards for total inventory, critical stock alerts, today's confirmed appointments, and total registered donors.
Inventory Status: Color-coded table (Green/Yellow/Red) showing stock by type and units expiring within 7 days.
Quick Actions: Large shortcut buttons for Search Donors, Manage Appointments, Manage Inventory, and View Reports.

B) Appointment Management Interface
Weekly Calendar View: 7-day grid showing numerical dates, appointment counts, and status-colored indicators.
Daily List: Chronological list of appointments featuring:
Donor ID, Name, and Blood Type (or "To Be Determine").
Status badges and a dropdown to update status (e.g., Checked-in, Completed).
Process Donation button for active appointments.
Donation Processing Modal: A focused window to confirm donor identity and "Complete Donation" to update the system records.

3.1.3 Donor Search Interface
Used by staff to locate and manage donor records efficiently.

Search & Filter: * Multi-parameter search (Name, ID, or Email)
Dropdown filters for Blood Type and Status (Active/Eligible/Deferred).
Donor Records Table: * Sortable columns for ID, Name, Blood Type, and Last Donation date.
Responsive design with row highlighting and a "View" action button.
Donor Detail Modal: Comprehensive profile view including contact info and donation statistics.

Summary of total units across all types.

Adequate (Green): 20+ units.
Low (Yellow): 10-19 units.
Critical (Red): < 10 units.
